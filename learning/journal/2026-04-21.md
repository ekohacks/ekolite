# 21 April 2026: Injection, Rebase, and the Cost of Silent Merges

> "Knowledge emerges only through invention and reinvention, through the restless, impatient, continuing, hopeful inquiry human beings pursue in the world, with the world, and with each other." Paulo Freire

A long day. Three threads ran in parallel, each one teaching something the others could not. Writing it down while it's still warm.

## Morning: the two Fastify instances

EKO-260 opened with a failing integration test. `createServer` built a Fastify, `start.ts` built a separate one and passed it to `WebSocketWrapper.create(fastify)`. The `/ws` route landed on one instance. `server.listen(3001)` ran on the other. The browser's `new WebSocket('ws://localhost:3001/ws')` gave `close code 1006`, because the route and the listener were on different objects.

The test was writing out a contract that the code refused to honour. `create()` with no arguments, said the test. The caller has no Fastify yet, because `createServer` has not built one. The old signature `create(fastify: FastifyInstance)` demanded a value nobody had. Which is why `start.ts` had invented a throwaway Fastify to satisfy the signature and the route ended up orphaned.

We bent the code to the test, not the other way round. `create()` now takes no arguments. A new method `attach(fastify)` lets the wrapper bind to Fastify later, when `createServer` has built one. Construction and binding, split into two moments. `start()` stays on the wrapper for raw mode, which boots its own standalone server and has no Fastify to attach to. Two modes, two verbs.

Manual test in the browser: socket opens. Tests green. Commit landed as `refactor(websocket): split wrapper construction from Fastify binding`. Pattern written up and posted on the PR.

## Midday: EKO-267 and the squash merge surprise

Ebuka's naming consistency PR went in with an empty body ("Work done"). Wrote a proper description and pushed it up. Then grep'd the branch for stale references from the renames. Code was clean. Docs were not. Diagrams and retro under `docs/websocket-architecture/` still described the current architecture using the old class names.

Updated five diagram files on the branch. Committed, pushed. While that was happening, both EKO-267 and EKO-260 got squash merged into main. My docs commit was already pushed to the branch but squash merges only take the PR's squashed commit. The docs update was left stranded on the merged branch, not on main.

Checked main. Old names still scattered across every diagram. The fix is ready as one commit on a defunct branch. Needs a follow-up PR to bring it onto main. Small thing, but a reminder that the timing of push versus merge matters, and squash merges discard anything that lands after the squash point.

## Afternoon: reviewing PR #24

Gochi's PR adopts ConfigurableResponse in `FileStorage` and `ScriptRunner`. Most of it is solid. The "mutation blocked when error configured" invariant is correctly enforced: the configured response is consumed before any store mutation, so an error aborts cleanly. The `trackChanges` event pattern matches MongoWrapper's shape, which makes the wrappers feel like siblings.

One real bug inside `StubbedFileSystem.exists`:

```ts
this.existsResponses?.next() as boolean;
const exists = this.store.has(name);
```

The cast is pure dead code. The return of `next()` is read and discarded, so the `as boolean` asserts nothing. Worse, the feature it implies (configurable exists override) never works. If a caller wrote `createNull({ exists: [true] })` hoping to override an empty store, their `true` would vanish. Only the error throwing side of the feature survived.

Two ways to resolve. Errors only, which means tightening the type to `Error[]` and dropping the dead cast. Or allow value override, which means using `next()`'s return and adding a test. Posted both on the PR and asked which was intended.

Gochi came back on Slack asking about the team position on `toScriptResult` and type casting. Written up a short stance: drop the helper, drop the string shorthand, require full `ScriptResult` objects. No `as` casts in the wrappers, treat them as a signal that the upstream type is too loose. Linked back to the PR review for the fuller argument.

## Late: the merge that taught us about rebase

Pulled main into EKO-265 and did a merge. All tests passed locally. Looked good.

Then tsc barked. Three separate errors, all from the same kind of conflict resolved silently by the merge tool:

- `fileStorage.ts` imported from `./output_tracker.ts`. Main had renamed it to `./outputTracker.ts`. Merge kept the old name.
- `scriptRunner.ts` referenced `ProcessRunnerInterface`. Main had renamed it to `ScriptRunnerInterface`. Merge kept the old name.
- The test file passed arrays to `createNull`, but `createNull`'s outer signature still read `Record<string, string>`. Merge took Gochi's tests but kept main's signature.

The third one is the loudest signal. Tests and production code pointing at different worlds, inside the same branch, with all tests green locally on each side but failing to typecheck when combined. The merge tool made the decision without asking. The inconsistency arrived in one silent blob.

Fixed the three issues, committed, everything clean.

A rebase would have stopped git at the first commit that clashed. It would have shown the diff at that exact point in history and said "this file was renamed upstream, decide now". It would have stopped again when a later commit tried to use the old interface name. Each conflict confronted on the commit that caused it, with the change still fresh.

The rule we can take from this: if a branch has been parallel to main for more than a day or two, rebase instead of merging. Merge hides decisions. Rebase makes you walk through each one. Both produce the same final tree, but one of them catches mismatches and the other lets them through.

## What I want to carry forward

Three patterns from today, each one bigger than the story it came from.

1. **Split construction from binding** when a dependency does not exist yet at the moment you would hand it over. `create()` plus `attach(x)` beats `create(x)` when `x` isn't available. The injection contract lives longer.

2. **A test that disagrees with the code is telling you something**. Do not default to adjusting the test. Ask which side is telling the truth. Usually it's the test.

3. **Merging a stale branch into main hides the decisions**. Rebasing surfaces them. Pick the workflow that makes the work visible.
