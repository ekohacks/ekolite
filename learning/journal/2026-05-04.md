# 4 May 2026: Silent Skip and the Question Behind It

> "Problem posing education affirms men and women as beings in the process of becoming." Paulo Freire

PR #35 today, Ebuka's 3.B.3, handle changed and removed. Two passes. The substantive comment landed in the first review, and the second pass confirmed he had taken both points cleanly. By the end the contract was symmetric, every successful mutation emits, no op mutations emit nothing, two short tests pinning the unknown id paths.

The PR itself is fine. What I want to keep on paper is a question that the review surfaced and that the merge does not actually answer.

## The shape of the asymmetry

When `changed` arrives for an id the client has never heard of, the original code did this:

```ts
case 'changed': {
  const existing = this.docs.get(msg.id);
  if (!existing) {
    return;
  }
  this.docs.set(msg.id, { ...existing, ...msg.fields });
  break;
}
```

When `removed` arrived for an id the client did not have, the original code did nothing of the kind. It called `Map.delete`, which silently no ops on missing keys, and then fell through to the emit at the bottom. So the contract was inconsistent. The fix Ebuka took was to short circuit `removed` the same way, which is the right call. Symmetric beats asymmetric. The tests now lock both branches into the same shape.

That part of the review did its job.

## The question behind the question

The thing I never got an explicit answer on is whether silent skip is what we actually want here, or whether we just chose it because it was the cheapest thing.

A `changed` for an unknown id, in a Meteor style sync protocol, means the client and server disagree about what the client has. Three honest readings of that gap:

1. **Silent skip.** The client carries on. Sync drifts until the next `added` brings the doc in or until a resync. Cheap, never crashes, but a real bug in the protocol can stay invisible for arbitrarily long.

2. **Throw.** The client refuses to keep going. Loud, makes the bug obvious during development, but breaks production for the user the moment any drift happens.

3. **Log a warning, behave like added.** Halfway. You preserve the data you do have, you flag the gap. A bit of complexity, debatable correctness if `msg.fields` is partial in ways that an `added` message would not be.

Ebuka chose silent skip and pinned it with tests. The tests document the behaviour. They do not document the reason.

That is the gap I want to come back to with him. Not as a code review note, since the PR has merged its way to a sensible default. As a conversation. Probably on the next call. Probably framed as: "we have silent skip locked in. What did you weigh when you picked that, and what would change your mind?"

## Why I am writing this down rather than reopening the PR

A second review pass to ask "but why" after he has already answered with code feels like moving the goalposts. He shipped working, tested, symmetric behaviour. Reopening to ask for a comment in the code that explains the choice would frame his merge as half done. It was not. It was a fine stopping point.

The journal is the right surface for this. The thing I want to know is not "is the code correct" (it is) but "did you think about this, and if you did, what was the case for the choice you made". That is a teaching question, not a review question. It belongs over coffee or on the call, not in a thread that the rest of the team is reading as a verdict.

## What I might ask, when the moment comes

A few framings worth holding loose. The point is not to pick the right one in advance. The point is to remember the question is there.

- What does silent skip cost us if it happens in production?
- What signal would we want, for ourselves, that it had happened?
- Is there a version of the protocol where this scenario is provably impossible, and would building toward that be a better use of time than instrumenting the symptom?

I do not know the answers. I do know that we shipped a default and we have not asked.

## What I carry forward

1. **A merge does not always answer the question that prompted the review.** Sometimes the dev picks the cheapest passing path because that is what TDD asks for. The path is right by the lights of the test. The question of whether the test should ask for something stronger is a separate conversation.

2. **The journal is the right surface for "what did you weigh when you chose X".** Those are teaching prompts, not blockers. Putting them on the PR turns dialogue into review, and the dev reads them as todos.

3. **Silent skip is the easiest default to pick and the hardest one to unpick later.** Once tests pin the behaviour, the cost of changing it goes up. Worth flagging now while the choice is still warm in the dev's head, even if I am flagging to myself.

## Update, later the same day

Had the conversation with myself instead of waiting for coffee. Two questions surfaced and pushed the silent skip thinking forward.

First, fail fast and restart. The pattern works only if four pieces are in place: an error boundary that catches the throw, a reload mechanism that is cheap and idempotent, transient state the user does not lose on reload, and a bounded retry budget so a permanent server bug does not cause an infinite restart loop. Without those four, throwing leaves the user with a dead tab and zero diagnosis. The framework today ships none of them. So throw on its own is worse than silent skip from the user's seat. If we want fail fast as the contract, we have to ship the contract, not just the throw.

Second, and the deeper one: is unknown id genuinely a bug or a legitimate race in the protocol we are modelling? Meteor's DDP allows messages to arrive out of order in some configurations. A `removed` followed by a `changed` for the same id is a perfectly normal artefact of network reordering. In that scenario the state IS consistent, just temporally noisy, and silent skip is exactly right. If we throw, we break valid sessions on transient races. If we model out of order delivery as a bug, throw is correct, silent skip is the bug. We have not decided which.

My honest read after sitting with it: this is a race, not a bug, in the protocol shape we are actually modelling. Silent skip is the correct behaviour. The gap is observability, not logic.

## Where this leaves us

Silent skip stays. The next move is the observer interface, not a `console.warn` patch and not a fail fast pivot.

Shape of the work:

- A pluggable observer that `ReactiveStore` and `Publications` call on every message lifecycle event (`applied`, `skipped`, `failed`).
- Default observer is no op, framework stays light. Consumers wire whatever they want: `console.warn`, Sentry, an OTel exporter, custom dashboards.
- The silent skip path becomes `observer.onMessage(msg, 'skipped')` instead of a bare return. Same observable behaviour from the framework's side, full visibility on the consumer's side.

OpenTelemetry is the right answer for production grade signal but the wrong layer to bake into the framework core. EkoLite is a teaching framework on a public repo, adding OTel as a dependency drowns the lesson. The clean version is a companion package, `ekolite-otel`, that wires standard OTel patterns against the observer interface. Anyone who wants observability gets a one liner. Anyone who does not, does not pay for it. That package is its own future story.

## What I carry forward

1. **The right answer to "no signal" is a hook, not a log line.** A `console.warn` patch in the framework would have answered today's symptom and made tomorrow's choices harder. The observer interface answers the symptom and leaves the consumer free.

2. **Pick the model before picking the behaviour.** Once I asked whether unknown id is a bug or a race, the silent skip question collapsed into a much smaller question. We had been arguing behaviour without agreeing on what model the behaviour was meant to satisfy.

3. **Shipping a default contract requires shipping its surrounding pieces.** Throwing without an error boundary, a reload, a state preservation story, and a retry budget is not fail fast. It is "the user sees a broken tab and we have no story for them." The contract is the four pieces, not the throw.
