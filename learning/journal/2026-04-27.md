The story still leaves the choice with the dev but does the work of surfacing the tension instead of leaving it as an exercise.

Whether that's an improvement or a slide back toward prescription, I'm honestly not sure. Worth watching how 3.B.2 lands when someone picks it up.

## Afternoon: caught myself posting without showing the draft

## The "out of scope" miss

The bigger lesson came out of a single bullet in that draft.

I'd written "client disconnect without unsubscribe — out of scope, but worth a Linear ticket". Sounded reasonable. The story file does scope itself to the unsubscribe message, so I let myself off the hook.

Read it again with the question "what is this story actually trying to deliver, and would it be delivered without disconnect handling". Definition of done says "no memory leaks on repeated subscribe/unsubscribe". OK. So what's the dominant way a subscription ends in production?

Spent a minute writing it down. Tab close. Refresh. Sleep. Wifi drop. Tab freeze when backgrounded too long. React strict mode double-mount. The polite `unsubscribe` message is the rare path. Disconnect is the dominant path. Shipping 3.A.4 without disconnect handling means shipping a system that handles the rare leak case correctly and the dominant leak case not at all.

So the "out of scope" framing wasn't out of scope of the work, it was out of scope of the **story's wording**, which is a different thing.

The story named itself "Unsubscribe Stops Updates", scoped its handler to the `unsubscribe` message, and put "no memory leaks" in its definition of done. Three sentences that quietly contradict. The dev who wrote against that spec built exactly what it asked for. The fault is upstream of him. The fault is in the title.

Updated the third bullet of the draft to say so honestly, and to hand the call back to Ebuka instead of pre-deciding it. Either fold disconnect teardown into this PR (data structure already supports it, dozen lines of code, one test) or split it into 3.A.5. His call.

## The duplicate sub id question

The other open thread from the review was the "dispose old first" choice the PR makes when a client subscribes twice with the same sub id. Started to write a short note explaining why that's the right call. Got asked to slow down and reason through it properly because the user (me again) didn't fully see why themselves.

Walked it. Sub ids are chosen and managed by the client. The server doesn't generate them, doesn't validate them, doesn't know what they mean. They're labels the client uses on its own side. When a duplicate arrives, the only thing it can mean is "this is what sub1 refers to now". The strict server replies "no, you already told me what sub1 was", which is making a claim about the client's state against the only entity that knows. That's polite-but-wrong. The tolerant server takes the new subscribe as authoritative and rebuilds.

The cost of tolerance is real. A client bug that reuses a sub id will produce subtle churn instead of crashing. The cost of strictness is paid every time, on every retry, every reconnect, every React strict-mode mount. Browsers are unpredictable. Tolerance is the right default for a real-time data layer.

Two decisions, same fault line. Disconnect teardown and duplicate sub id are the same question wearing different hats. The question is "who owns subscription state", and the answer is "the client, always". Both decisions follow from there. The server is downstream of the client and absorbs whatever reality the client presents.

That through-line is what I sat down and wrote up after the review went out.

## The deep-dive piece

`learning/unsubscribe-is-the-rare-path.md`. Roughly 1,400 words.

Thesis: a story named after a message handler delivers a message handler. A story named after a goal delivers the goal. The 3.A.4 story called itself "Unsubscribe Stops Updates" and the PR ships a watcher leak in the dominant case because the title decided the scope before any code was written.

Pulled both stance questions into the same arc. Server is downstream of the client. Subscription state is owned by the client. Disconnect is the rare path's twin, duplicate id is the same question. Closed with a Freirean prompt asking the reader to look at the next ticket in their backlog and check whether the title names the user-visible outcome or the function they're about to write.

Whether that piece holds up depends on whether the audience reads it as one of those "I've worked it out, here's what you should do" articles or as the genuine "I caught this in my own draft, here's the tooth I had to pull". Tried to write the second one. Drafted it inline before saving the file. Will sit with it before pressing publish through whatever the dojo's content channel ends up being.

## What I carry forward

1. **Story titles decide scope.** The 3.A.4 spec is honest about what it's scoped to, and that scope ships a leak. Before the next story spec goes out, write the title twice. Once as "handle the unsubscribe message" and once as "subscriptions don't leak". Look at both and pick the one that names the outcome. The first costs less to write. The second costs less to live with.

2. **"Out of scope" is a stance.** Whenever I find myself writing those three words in a review, the next thing to write is which scope, defined by whom. If the answer is "scope as named in the story title" and the story title names a piece of the implementation rather than the outcome, the framing is upstream of me and I should say so plainly instead of inheriting it.

3. **Show the draft, every time.** Reviews, PR comments, Slack, Linear, anything that lands under my name on a public surface goes through a deliberate read before it ships. No exceptions, no "this one is fine". Saved this as a memory rule so I stop relitigating it on each turn.

4. **Reason through the things I'm asking other people to reason about.** The duplicate sub id walk happened because I was about to post a one-line "curious why you picked tolerant" without being able to articulate why I myself thought tolerant was right. The note that came out of slowing down is now more useful to Ebuka than the original one-liner would have been, and more honest about where my own reasoning sat.
